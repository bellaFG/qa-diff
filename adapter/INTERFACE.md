# Adapter Interface

An adapter tells the QA pipeline how your project works — file layout, test commands, auth patterns, etc.

Adapters are generated by `/qa-setup` and saved to `adapter/project.md`. You can also create one manually following this contract.

---

## REQUIRED Sections

### `file_patterns`

Map your source directories to pipeline categories. The pipeline uses these to categorize changed files.

```yaml
file_patterns:
  model: "app/models/**/*.rb"
  endpoint: "app/controllers/**/*.rb"
  service: "app/services/**/*.rb"
  view: "app/views/**/*.erb"
  script: "app/javascript/**/*.{js,vue}"
  migration: "db/migrate/*.rb"
  helper: "app/helpers/**/*.rb"
  job: "app/jobs/**/*.rb"
  config: "config/**/*.{rb,yml}"
  test: "spec/**/*_spec.rb"
```

You may omit categories that don't apply (e.g., `migration` for a Go project). At minimum, provide `model`, `endpoint`, and `test`.

Also provide source directories for diff scoping:

```yaml
source_dirs: "app/ lib/ config/"
```

### `auth_pattern`

How to set up authentication in tests. Must include `setup` (how the main test user is created and logged in) and `cross_tenant` (how to create a second user from a different tenant).

```yaml
auth_pattern:
  setup: |
    user = create_user()
    authenticate(user)
  cross_tenant: |
    other_user = create_user(tenant: "other")
    foreign_resource = create_resource(owner: other_user)
  notes: "Any special considerations for auth in tests"
```

### `idor_detection`

How to find insecure direct object references. Must include a scan command, safe patterns, and vulnerable patterns.

```yaml
idor_detection:
  scan_command: "grep -n 'params\\[:.*_id\\]' {file}"
  safe_patterns:
    - "current_user.association.find(params[:id])"
    - "policy_scope(Model).find(params[:id])"
  vulnerable_patterns:
    - "Model.find(params[:id])"
    - "Model.where(id: params[:id]).first"
```

### `test_execution`

How to create and run tests. Must include the test command and the temp spec directory structure.

```yaml
test_execution:
  command: "bundle exec rspec {specs} --require ./spec/rails_helper --format documentation"
  coverage_flag: "COVERAGE=true"
  temp_dir: "/tmp/qa-specs"
  structure:
    - endpoints/
    - models/
    - load/
  file_suffix: "_spec.rb"
  framework: "rspec"
```

### `assertions`

Patterns the Sentinel H1-H10 uses to detect spec quality issues. Must provide patterns for each detection type.

```yaml
assertions:
  mock_assertion_pattern: "have_received"
  last_record_pattern: "Model\\.last|Model\\.first"
  weak_assertions:
    - "be_present"
    - "be_truthy"
    - "be_nil"
    - "be_falsy"
  status_only_pattern: "have_http_status"
  mock_syntax: "allow\\(.+\\)\\.to receive"
  auth_mock_pattern: "allow\\(controller\\)\\.to receive\\(:current_user\\)"
  exact_value_assertion: "eq("
```

### `factory_pattern`

How to create test data with real persistence.

```yaml
factory_pattern:
  create: "create(:factory_name, attributes)"
  build_forbidden: "build_stubbed"
  discovery_command: "grep -rl 'factory :{model_name}' spec/factories/ | head -5"
  notes: "Always use create() for real persistence, never build_stubbed()"
```

### `spec_type_mapping`

Which spec type to use for each source category.

```yaml
spec_type_mapping:
  model: "request spec via HTTP; model spec only if no endpoint exists"
  endpoint: "controller/request spec"
  helper: "integration spec within endpoint context"
  job: "job spec with real execution (perform_now)"
  view: "endpoint spec verifying response body"
```

## OPTIONAL Sections

### `coverage`

How to measure test coverage.

```yaml
coverage:
  tool: "simplecov"
  result_path: "coverage/.resultset.json"
  extraction_script: |
    # Script that takes file paths and outputs coverage percentages
    # for lines changed in the diff
```

### `security_tools`

SAST and security scanning tools available.

```yaml
security_tools:
  - name: "brakeman"
    command: "bundle exec brakeman --no-progress --quiet -w 2"
    categories: "SqlInjection,RemoteCodeExecution,CrossSiteScripting"
  - name: "rubocop-security"
    command: "bundle exec rubocop --only Security"
```

### `mutation_testing`

Mutation testing tool configuration.

```yaml
mutation_testing:
  command: "bundle exec mutant run --include app --require rails_helper --use rspec \"{class}\" --since origin/main"
  fallback: "property-based testing with random inputs"
  available: true
```

### `load_testing`

Load testing templates and tools.

```yaml
load_testing:
  benchmark_template: |
    # Inline benchmark script template
  http_template: |
    # HTTP load test script template
  query_counter_template: |
    # N+1 detection script template
```

### `lint`

Linting tools for code style.

```yaml
lint:
  command: "bundle exec rubocop {files}"
```

### `next_steps`

Project-specific commands to show in the report's "Next steps" section.

```yaml
next_steps:
  - "/check-ci"
  - "/review-kody"
  - "/pr-ready"
```

---

## Validation

When the QA engine loads an adapter, it validates:

1. All REQUIRED sections are present
2. `file_patterns` has at least `model`, `endpoint`, and `test`
3. `auth_pattern` has both `setup` and `cross_tenant`
4. `idor_detection` has `scan_command`, `safe_patterns`, and `vulnerable_patterns`
5. `test_execution` has `command`
6. `assertions` has all 7 patterns

Missing required section → STOP with clear error message telling the user to run `/qa-setup` or manually add the section.
