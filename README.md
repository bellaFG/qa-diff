# QA Diff

An adversarial QA pipeline for [Claude Code](https://claude.ai/claude-code) that works with **any language and framework**.

QA Diff analyzes your PR diff, writes integration tests that try to break your code, hunts for IDOR vulnerabilities, and produces a structured report — all without modifying your production code.

## What It Does

- **8-phase pipeline**: Discovery, Planning, Execution, Classification, Load Testing, Report, Triage, Persist
- **IDOR detection**: Scans endpoints for unscoped parameter lookups (Insecure Direct Object Reference)
- **Adversarial testing**: Tries null, empty, invalid, XSS, and cross-tenant inputs
- **Sentinel H1-H10**: Self-audits spec quality before execution (catches mock-only tests, weak assertions, etc.)
- **Boundary value mapping**: Systematically tests edge cases for every new parameter
- **Mutation testing**: Validates that tests actually catch bugs (not just green-bar theatre)
- **CI integration**: GitHub Actions gate that enforces QA before merge

## Quick Start

### 1. Copy into your project

```bash
# Option A: Copy the directory
cp -r qa-diff/ /path/to/your/project/qa-diff/

# Option B: Git submodule
git submodule add https://github.com/YOUR_USER/qa-diff.git qa-diff
```

### 2. Run setup

```
/qa-setup
```

The setup wizard will:
- Auto-detect your language, framework, and test runner
- Ask about your auth model and multi-tenancy setup
- Generate `adapter/project.md`, `examples/project.md`, and `config.yml`
- Optionally install CI gates

### 3. Run QA on your PR

```
/qa-diff          # current branch vs main
/qa-diff 123      # PR #123
/qa-diff feature/x  # specific branch
```

## How It Works

```
/qa-diff [PR|branch]
  |
  v
Phase 1: Discovery
  - Analyze git diff, categorize files, detect IDOR patterns
  - Classify PR type (SECURITY > LOGIC > ENDPOINT > VIEW > CONFIG)
  - Calculate budget based on PR size
  |
  v
Phase 2: Planning  <-- developer confirms plan
  - Build boundary value map for each parameter
  - Map test scenarios (regression, IDOR, adversarial, edge cases)
  |
  v
Phase 3: Execution  (zero interruptions)
  - Write integration specs in /tmp/qa-specs/
  - Self-audit with Sentinel H1-H10
  - Run tests with coverage
  - Mutation testing (for calculation-heavy code)
  |
  v
Phase 4-6: Classify + Load Test + Report
  - Classify findings by type and severity
  - Optional load testing for performance branches
  - Generate structured report with verdict
  |
  v
Phase 7: Triage  <-- developer decides per finding
  - Present all findings at once
  - Developer chooses: [C]oder fixes / [I]ssue card / [X] ignore
  |
  v
Phase 8: Persist
  - Save report to qa-reports/
  - Cleanup temp specs
```

## Supported Stacks

QA Diff is framework-agnostic. The `/qa-setup` wizard has built-in detection for:

| Language | Frameworks | Test Runners |
|----------|-----------|--------------|
| Ruby | Rails | RSpec, Minitest |
| Python | Django, FastAPI, Flask | pytest, unittest |
| JavaScript/TypeScript | Next.js, Express, Vue, React | Jest, Vitest, Mocha |
| Go | Gin, Echo, Fiber, Chi | go test |
| Rust | Actix, Axum, Rocket | cargo test |
| Java/Kotlin | Spring, Quarkus | JUnit |
| Elixir | Phoenix | ExUnit |
| PHP | Laravel, Symfony | PHPUnit |

For other stacks, the wizard asks for your patterns and generates a custom adapter.

### Findings
**Bug 1** · payments_controller.rb:42 · Vulnerability · HIGH · CODER
IDOR: `Payment.find(params[:id])` without tenant scope — user A can access user B's payments
Expected: `current_user.payments.find(params[:id])`

**Improvement 1** · payment.rb:15 · FRAGILE · MEDIUM · IGNORED

## Verdicts

| Verdict | Meaning |
|---------|---------|
| **APPROVED** | Zero open findings |
| **APPROVED_WITH_PENDING** | Findings exist but developer chose to defer/ignore in triage |
| **BLOCKED** | Environment issues or >50% of files could not be tested |

## Requirements
